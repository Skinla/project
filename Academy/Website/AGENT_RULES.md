## Правила для dev‑агента / агентства

Цель: разработать серверный модуль для конструктора сайта (см. `PROMPT.md`) **без изменения каталога Bitrix24**.

### 1) Источники правды (что читать и чему верить)

Порядок чтения:
1) `PROMPT.md` — контракт и поведение модуля (эндпоинты, кеш, ошибки, ограничения).
2) `project plan.md` — ТЗ/макеты и общий контекст проекта.
3) `FIELD_MAPPING.tsv` — таблица “поле → источник → зачем → где используется” (открывается в Excel).
4) `REGISTRY_RULES.md` — точные правила по `property122` (“Реестр”).
5) `MISSING_DATA.md` — список дыр/решений по данным и фоллбеки.
6) `DEPLOYMENT.md` — как хранить секреты/деплоить на Bitrix Box.

Данные‑примеры (для проверки контрактов):
- `webhook_check/catalog.product.get_18.pretty.json` — полный пример товара, где видно `property122 = "ФИС ФРДО"` и требования списком.
- `webhook_check/variability.report.pretty.json` — вариативность по разделам (важно для `property122` и пустых полей).

### 2) Жёсткие запреты
- **Нельзя менять каталог Bitrix24**: не добавлять/не править свойства/поля/индексы/шаблоны URL.
- **Нельзя коммитить секреты**:
  - реальный `bitrix.access.env` держать вне репозитория (см. `DEPLOYMENT.md`).
- **Не использовать**:
  - картинки товара
  - “старую цену”
  - описания (`previewText/detailText`) — по решению, т.к. почти всегда пусто.

### 3) Что обязательно реализовать в модуле
- Реализовать **3 блока** (как в ТЗ/макетах) + API для них:
  - `academyprofi.search` → `/academyprofi-catalog/search?q=...`
  - `academyprofi.catalog` → `/academyprofi-catalog/products?start=...`
  - `academyprofi.detail` → `/academyprofi-catalog/product?id=...` (см. `PRODUCT_DTO.md`)
- Кеш (TTL 5–15 минут) + логирование (cache hit/miss + ошибки Bitrix).

Out of scope:
- любая серверная “логика заявки” (лиды/вебхуки). Заявка — это form‑block на странице.

### 4) Фиксированные решения (чтобы не задавать вопросы по кругу)
1) **Откуда берём `productId` для detail‑block**:
   - приоритет: URL текущей страницы (`#product=<id>` или `?productId=<id>`)
   - fallback: настройки блока (дефолтный `productId` для демо/превью)
2) **Якорь на форму**:
   - по умолчанию `#request` (настраивается)
3) **Где живут assets блоков**:
   - рядом с приложением на `bitrix.axiom24.ru` и подключаются через `manifest.assets.*`
4) **Регистрация блоков в конструкторе**:
   - при установке local app `install.php` вызывает `landing.repo.register` для всех блоков (по версии приложения)

Пока эти решения не даны — хранить их в настройках модуля:
- `MODULE_SETTINGS.example.json`

### 5) Как обновлять документацию
- Любое найденное “новое знание” (мэппинг/правило/решение) обновлять в:
  - `FIELD_MAPPING.tsv` (таблица),
  - и при необходимости в `PROMPT.md` (контракт модуля).
