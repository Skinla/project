# План проекта: Вариант A (каталог Битрикс24)

## 1. Цель
Сделать **детальную карточку услуги** как **встраиваемый блок** на странице, используя данные товара из каталога Битрикс24. На этой же странице рядом размещается **встраиваемый form‑block** (“Оставить заявку”), без серверной логики заявки в модуле. Внешний сервер допустим для API/кэша.

## 2. Исходные данные
- Платформа: Битрикс24 (облако), тариф Professional, Маркет доступен.
- Товары уже хранятся в каталоге Битрикс24.
- Детальная карточка — **встроенный блок**. `productId` определяется:
  - приоритет: из URL текущей страницы (`#product=<id>` или `?productId=<id>`)
  - fallback: из настроек блока (для демо/превью в редакторе)
- Кнопка “Оставить заявку” ведёт якорем на форму на этой же странице (по умолчанию `#request`).

## 3. Архитектура (выбранный подход)
Основной источник данных — каталог Битрикс24.

### 3.1. Доступ к данным
Два возможных режима:
1) Прямой доступ к данным через REST API Битрикс24.
2) Внешний сервер‑прокси (рекомендуется), который:
   - получает данные из Битрикс24 по API,
   - кэширует результат,
   - отдает данные сайту без раскрытия токенов,
   - решает проблему CORS и лимитов API.

### 3.2. Встраивание на сайт
Кастомные HTML/CSS/JS блоки в конструкторе сайта Битрикс24.

#### 3.2.1. Набор блоков (по ТЗ)
1) **Поиск** (`academyprofi.search`): поле + dropdown до 8 результатов, старт с 3 символов.
2) **Каталог** (`academyprofi.catalog`): карточки + пагинация (3 видно, 5 всего).
3) **Деталка** (`academyprofi.detail`): детальная карточка услуги (см. поля ниже).

### 3.3. Ограничения по каталогу (жёсткое правило)
- **Каталог Bitrix24 считаем read-only**: не добавляем/не меняем свойства/поля, не заводим новые “служебные” поля под URL/ФРДО/картинки и т.п.
- Всё, чего не хватает по ТЗ, делаем:
  - нормализацией данных на стороне приложения (third‑party сервер / модуль),
  - производными полями/эвристиками,
  - хранением индексов/кеша на стороне приложения.

## 4. Структура данных (поля товара)

- Учебные часы

### 5.2. Пагинация

- При наведении на следующую страницу — подчеркивание (#f2f2f2).

### 5.3. Поиск

- Дизайн и анимация поиска оставляются как в примере.

## 6. План работ по этапам

### Этап 1. Подготовка каталога
1) Уточнить и добавить свойства товара в каталоге.
2) Проверить заполнение всех обязательных полей.
3) Убедиться, что у каждой услуги есть URL детальной страницы.

Примечание (актуальное правило):
- Этап “добавить свойства/URL в каталог” **не выполняем** (каталог не меняем). Если чего-то нет — обрабатываем в модуле/на сайте (параметры URL/попап).

### Этап 2. API (если используем внешний сервер)
1) Настроить получение товаров через REST API Битрикс24.
2) Реализовать эндпоинты для блоков:
   - GET /products?start=... (каталог/карточки + пагинация)
   - GET /search?q=... (поиск)
   - GET /product?id=... (детальная карточка)
3) Кэшировать ответы (например 5–15 минут).
4) Вернуть нормализованную структуру данных для фронта.

### Этап 3. Фронт‑блоки на сайте
1) Блок **поиска** (HTML/CSS/JS).
2) Блок **каталога** (HTML/CSS/JS).
3) Блок **детальной карточки услуги** (HTML/CSS/JS).
4) Блок **формы заявки** (form‑block) на этой же странице (вне этого приложения).
5) JS‑логика:
   - поиск/каталог выставляют `#product=<id>` (или `?productId=<id>`) при клике по карточке/результату,
   - detail‑block читает `productId` из URL и подтягивает данные,
   - кнопка “Оставить заявку” делает scroll/jump на якорь формы (например `#request`).

### Этап 4. Интеграция в Битрикс24
1) Вставка HTML/CSS/JS в конструктор.
2) Проверка работы на опубликованном сайте.

### Этап 5. Тестирование
1) Проверить поиск:
   - от 3 символов,
   - максимум 8 элементов,
   - скролл списка,
   - переход на карточку по клику.
2) Проверить сетку:
   - 3 видимые колонки, 5 всего,
   - корректная прокрутка/пагинация,
   - hover‑рамка.
3) Мобильная адаптация.

## 7. Критерии приемки

- “Оставить заявку” переводит к форме на этой же странице (якорь).
- Данные берутся из каталога Битрикс24 (через API/прокси), токены на фронт не попадают.

## 9. REST методы (каталог)
Рекомендуемый набор (торговый каталог):
- catalog.product.list — список товаров.
- catalog.product.get — карточка товара по ID.
- catalog.productProperty.list — список свойств товаров.
- catalog.section.list — разделы каталога.
- catalog.price.list — цены товаров (если цена не хранится в самом товаре).

Примечание: crm.product.list устарел, использовать только если каталог работает через CRM-товары.

### 10.2.1. Scope (метод `scope`)
Фактически доступные scope по вебхуку:
- `bizproc`, `catalog`, `crm`, `disk`, `documentgenerator`, `lists`, `rpa`

Примечания:
- `user.current` по этому вебхуку возвращает `insufficient_scope` (не блокирует работу с каталогом).
- Несмотря на отсутствие `sale` в `scope`, метод `catalog.price.list` фактически отрабатывает (пример: productId=7896).

### 10.2.2. Проверка методов каталога (фактические ответы)
Проверено, что методы отвечают:
- `catalog.product.list` (filter: `iblockId=14`)
- `catalog.product.get` (пример: `id=7896`, `iblockId=14`)
- `catalog.productProperty.list` (filter: `iblockId=14`)
- `catalog.section.list` (filter: `iblockId=14`)
- `catalog.price.list` (пример: `productId=7896`)

### 10.2.3. Важные особенности API (на которые опираемся в реализации)
- В `catalog.product.list` **обязательно** включать `iblockId` в `select` (иначе ошибка вида: “Required select fields: iblockId”).
- Пагинация `catalog.product.list`:
  - параметр `start` (0, 50, 100, …)
  - `total/next/time` находятся на верхнем уровне ответа (не в `result`)
  - фактический `total` по `IBLOCK_ID=14`: **2004**
- Поиск по подстроке в `catalog.product.list`:
  - работает фильтр **`%name`**
  - `*name` у текущего портала не дал результатов (не использовать)
