## Чего не хватает по ТЗ (и что нужно уточнить/найти в текущих данных)

Правило: **каталог Bitrix24 не меняем**. Значит “чего не хватает” = либо поля реально отсутствуют в текущем каталоге, либо заполнены плохо, либо требуют нормализации на стороне модуля.

Данные ниже собраны по фактическому скану всего каталога `IBLOCK_ID=14` (2004 товара) на 2026‑01‑24.

### 1) Отсутствует как данные (в текущем каталоге нечего взять)

- **Описание для детальной карточки**
  - `previewText`: заполнено 0 / 2004
  - `detailText`: заполнено 16 / 2004 (99.2% пусто)
  - **Решение**: описание **не используем** (по твоему решению).

### 2) Есть, но заполнено не на 100% (нужно решить fallback/поведение)

- **Учебные часы / объём курса** — `property112`
  - пусто у 39 / 2004 (1.9%)
  - **fallback**: скрывать строку “уч. часы”, если нет значения.

- **Реестр** — `property122`
  - пусто у 22 / 2004 (1.1%)
  - **fallback**: скрывать “Реестр”, если `-`/пусто.

- **Формат обучения** — `property124`
  - пусто у 32 / 2004 (1.6%)
  - **fallback**: скрывать блок.

- **Требования к клиенту** — `property126`
  - пусто у 6 / 2004 (0.3%) + часто встречается строка-заглушка `-` (считается пустым)
  - **нормализация**: превращать строку в массив (split по `\\n`, `;`, `,`) и чистить мусор; если пусто — скрывать список.

- **Требуемый уровень образования** — `property128`
  - пусто у 15 / 2004 (0.7%)
  - **fallback**: скрывать.

- **Раздел** — `iblockSectionId`
  - пусто у 1 / 2004 (пример: товар `7896`)
  - **fallback**: “Без раздела” или скрыть фильтр.

### 3) Есть, но требует договорённости, как это показывать

- **Вид услуги** — `property106` (enum)
  - заполнено 2004 / 2004
  - **нужно**: утвердить, что это и есть “Вид услуги” из макета.

- **Реестр / ФИС ФРДО** — `property122`
  - `property122` в реальных данных может содержать значение `"ФИС ФРДО"` (пример: товар `id=18`).
  - **Решение**: показываем строку “Реестр” и значение; отдельное поле “ФИС ФРДО” на UI не нужно — это просто одно из значений реестра.

- **Цена “от …”**
  - Розничная: `property152` заполнено 2004 / 2004 (Money `"N|RUB"`)
  - Партнёрская: `property154` заполнено 2004 / 2004
  - **нужно**: на UI используем `property152` (уже договорились), форматируем в строку `от N ₽`.
  - “Старая цена” **не нужна** (по твоему решению).

### 4) Детальная страница (URL/навигация) — нужно утвердить механику

Так как каталог менять нельзя и поля URL нет:
- “деталка” открывается как страница с параметром (`#product=<id>` или `?productId=<id>`) и подтягивает детали по API модуля.

Факт реализации:
- фронт (JS блока) использует hash/query как источник `productId` (по правилу “URL → fallback”)
- для демо есть fallback id в настройках (`ui.detail.productId`)
