## Описание

Этот репозиторий содержит один обработчик событий Bitrix, который:

- отслеживает добавление сообщений в чат (модуль `im`, событие `OnAfterMessagesAdd`);
- фильтрует сообщения по маркеру `SYSTEM WZ`;
- пишет запись в Журнал событий (EventLog);
- пытается определить `LEAD_ID` по чату открытых линий (imopenlines) через `CRM_ACTIVITY_ID`;
- запускает бизнес-процесс (bizproc) на найденном лиде;
- защищается от дублей запусков БП через файловую блокировку на 10 секунд.

Код находится в `local/php_interface/handlers/message_add.php`.

## Структура

- `local/php_interface/handlers/message_add.php` — обработчик `im:OnAfterMessagesAdd`.
- `local/php_interface/handlers/cache/` — директория для lock-файлов (создаётся на лету).

Примечание: в репозитории нет `init.php`/модуля/инсталлятора — подключение обработчика должно быть сделано на стороне Bitrix-проекта.

## Поток выполнения (message_add.php)

### 1) Регистрация обработчика

Регистрируется обработчик события:

- модуль: `im`
- событие: `OnAfterMessagesAdd`
- сигнатура: `function($messageId, $messageFields)`

### 2) Фильтрация по маркеру

- подключает модули `im` и `crm` (если не подключились — выход);
- берёт текст из `$messageFields['MESSAGE']`, trim, пустой — выход;
- проверяет текст регэкспом:

`/(^|\s)===[\s\S]*SYSTEM WZ[\s\S]*===/i`

Если маркер не найден — выход.

### 3) Логирование “маркер пойман”

Нормализует текст:

- заменяет переносы строк на ` \n ` (для читаемости в журнале),
- режет длину до ~1900 символов,

и пишет запись в EventLog:

- `SEVERITY = INFO`
- `AUDIT_TYPE_ID = WZ_SYSTEM_MARKER`
- `MODULE_ID = im`
- `ITEM_ID = <messageId>`

### 4) Определение лида по чату (Открытые линии)

Если `CHAT_ID > 0` и есть модуль `imopenlines`:

- берёт последнюю сессию по чату:
  - `Bitrix\ImOpenLines\Model\SessionTable::getList(... filter CHAT_ID, order ID DESC, limit 1)`
- из сессии берёт `CRM_ACTIVITY_ID`;
- по `CRM_ACTIVITY_ID` читает биндинги:
  - `Bitrix\Crm\ActivityBindingTable::getList(filter ACTIVITY_ID)`
- ищет binding, где `OWNER_TYPE_ID === CCrmOwnerType::Lead`;
  - тогда `LEAD_ID = OWNER_ID`.

Если лид не найден — пишет WARNING `WZ_NO_LEAD_FOR_CHAT` (только в ветке, где сессия найдена).

### 5) Защита от дублей (10 секунд на LEAD)

Перед запуском БП делается дедупликация через lock-файл:

- директория: `local/php_interface/handlers/cache`
- файл: `wz_bp_lead_<LEAD_ID>.lock`
- содержимое: UNIX timestamp последнего запуска

Алгоритм:

- пытается открыть `fopen($lockFile, 'c+')`;
- пытается взять эксклюзивный flock неблокирующе с таймаутом:
  - до 5 попыток по 100мс (≈0.5 сек максимум);
- читает timestamp из файла;
- если прошло \(\ge 10\) секунд — разрешает запуск и перезаписывает timestamp.

Периодически чистит старые lock-файлы:

- вероятность 20% на вход (`mt_rand(1, 5) === 1`);
- удаляет lock’и старше 60 секунд;
- ограничение: максимум 100 удалений за проход.

**Fallback-логика**: если с директорией/файлом/блокировкой проблемы — БП запускается всё равно (логируются WARNING-события), чтобы “лучше отправить, чем пропустить”.

### 6) Запуск бизнес-процесса

Если найден `LEAD_ID > 0` и подключён модуль `bizproc`, и защита от дублей разрешила запуск:

- `bpTemplateId = 446` (жёстко задано в коде)
- `documentId = ['crm','CCrmDocumentLead','LEAD_<LEAD_ID>']`
- `wfParams = []` (есть закомментированный пример передачи параметров)

Запуск: `CBPDocument::StartWorkflow(...)`

- при ошибках старта — `WZ_BP_START_ERROR` (SEVERITY ERROR)
- при успехе — `WZ_BP_STARTED` (SEVERITY INFO)
- если дедуп сработал — `WZ_BP_SKIPPED` (SEVERITY INFO)

## EventLog: audit-типы

Файл пишет события с `AUDIT_TYPE_ID`:

- `WZ_SYSTEM_MARKER` — маркер найден, обработчик продолжает работу
- `WZ_CACHE_DIR_ERROR` — не удалось создать `cache/`
- `WZ_INVALID_LEAD_ID` — некорректный `LEAD_ID`
- `WZ_LOCK_FILE_ERROR` — не удалось открыть lock-файл
- `WZ_LOCK_WRITE_ERROR` — не удалось записать timestamp в lock-файл
- `WZ_LOCK_EXCEPTION` — исключение при работе с блокировкой
- `WZ_LOCK_TIMEOUT` — не удалось взять flock за ~0.5 сек
- `WZ_BP_START_ERROR` — ошибки при `StartWorkflow`
- `WZ_BP_STARTED` — БП запущен
- `WZ_BP_SKIPPED` — БП пропущен из-за дедупа
- `WZ_NO_LEAD_FOR_CHAT` — не удалось определить лид по чату
- `WZ_BP_START_EXCEPTION` — исключение в ветке резолва/старта БП
- `WZ_HANDLER_EXCEPTION` — исключение верхнего уровня обработчика

## Конфигурация/рычаги для изменений (что обычно правят)

- **Маркер/условие запуска**: регэксп поиска `SYSTEM WZ`.
- **Резолв CRM-сущности**: сейчас ищется только `LEAD` через `CRM_ACTIVITY_ID` и биндинги.
- **Шаблон БП**: `bpTemplateId = 446`.
- **Окно дедупликации**: 10 секунд.
- **Хранилище дедупа**: файловая система `handlers/cache` (важно для кластеров).
- **Входные параметры БП**: сейчас пусто (`wfParams = []`).

## Ограничения

- Дедупликация через файлы **не защищает** от дублей между разными web-узлами (если нет общего FS).
- Если `SessionTable` не вернёт сессию по чату, обработчик после `WZ_SYSTEM_MARKER` фактически “тихо” ничего не сделает (без отдельного лога про отсутствие сессии).
