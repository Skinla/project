# Описание проекта `crmmarketing25`

## Состав проекта (актуальные файлы)

- `deal_webhook_handler.php` — основной webhook-обработчик: получает `deal_id`, читает сделку из облака, создает лид в коробке, запускает роботов/БП, пишет маппинг.
- `deal_webhook_config.php` — конфигурация интеграции: webhook URL, поля, ID списков/чата, тексты уведомлений, пути к файлам.
- `bitrix_init.php` — минимальная инициализация Bitrix для webhook-скрипта (`prolog_before.php`, модули `crm`/`iblock`).
- `deal_lead_pairs.json` — хранилище соответствий `deal_id -> lead_id` (идемпотентность).
- `tests/prod_precheck.php` — precheck прод-среды (права/модули/доступность webhook).
- `tests/prod_test_lead_792042.php` — тестовый прогон боевого обработчика для сделки `792042`.
- `prod_precheck.php` — совместимая обертка, запускает `tests/prod_precheck.php`.
- `prod_test_lead_792042.php` — совместимая обертка, запускает `tests/prod_test_lead_792042.php`.
- `PROJECT_DESCRIPTION.md` — текущая документация по проекту.
- `crmmarketing25.zip` — архив проекта.

## Какие папки/файлы создаются автоматически

- `var/deal_to_lead_webhook.log` — основной лог обработчика (создается автоматически при первой записи).
- `deal_lead_pairs.json.lock` — lock-файл на время записи маппинга `deal_id -> lead_id` (используется для защиты от гонок).
- `deal_lead_pairs.json.tmp` — временный файл при атомарном сохранении маппинга (создается/переименовывается в процессе записи).
- Каталог `var/` для логов создается автоматически, если отсутствует.

## Назначение

Проект синхронизирует данные между:

- облачным порталом `crmmarketing25.bitrix24.ru` (источник сделок),
- коробочным порталом Bitrix (приемник лидов и связанной бизнес-логики).

Основной сценарий: при изменении сделки в облаке выполняется обработчик, который на стороне коробки создает лид и сохраняет связь `deal_id -> lead_id`.

---

## Общая логика работы

1. Облачный Bitrix24 отправляет webhook (обычно `ONCRMDEALUPDATE`) на `deal_webhook_handler.php`.
2. Обработчик извлекает `deal_id` из разных форматов входных данных (`$_REQUEST`, `$_POST`, `$_GET`, raw body).
3. Проверяется, не была ли сделка уже обработана по `deal_lead_pairs.json`.
   - для повторной обработки по GET (ссылка из чата) возвращается HTML-страница с парой `сделка -> лид` и кнопкой "Открыть лид";
   - для webhook/POST остается JSON-ответ `already_processed`.
4. Если не обработана, скрипт запрашивает детали сделки из облака через REST (`crm.deal.get`).
5. По названию сделки ищется элемент в инфоблоке 54 коробки.
6. Формируется payload лида (ФИО, телефон, источник, город, ответственный, наблюдатели, комментарии).
7. Лид создается во внутреннем API коробки через `CCrmLead::Add` с передачей `CURRENT_USER`.
8. Сразу после создания лид явно запускает CRM-автоматизацию:
   - роботы через `\Bitrix\Crm\Automation\Starter::runOnAdd()`,
   - бизнес-процессы через `\CCrmBizProcHelper::AutoStartWorkflows(...)`.
9. Фиксируется результат в `deal_lead_pairs.json` (с lock при записи, чтобы избежать гонок при параллельных webhook).
10. В случае ошибок пишется лог и, для части кейсов, отправляется уведомление в чат Bitrix (`CIMMessenger::Add`).

## Что именно используется: REST API или внутренний API

Используется **гибридная схема**:

- **Облачный REST API Bitrix24** (вебхук) — для чтения данных из облака.
- **Внутренний API коробки Bitrix** (legacy + частично D7) — для создания/обогащения сущностей.

Это не "только API одного типа", а смешанный интеграционный поток между двумя контурами Bitrix.

---

## Создание сущностей в Bitrix (как сделано в проекте)

### 1) Создание лида (коробка)

Лид создается внутренним методом:

- `new CCrmLead(false)`
- `CCrmLead::Add($arFields, true, ['REGISTER_SONET_EVENT' => 'Y', 'CURRENT_USER' => <id>])`

Ключевые поля при создании:

- стандартные CRM поля (`TITLE`, `NAME`, `SECOND_NAME`, `LAST_NAME`, `STATUS_ID`, `SOURCE_ID`, `ASSIGNED_BY_ID`, `COMMENTS`),
- мультиполе телефона через `FM[PHONE]`,
- пользовательские поля `UF_CRM_*`.
- `SOURCE_ID` сначала берется из связанного элемента списка источников, а если не найден — используется fallback из конфига `lead.default_source_id` (сейчас `UC_HJOQ1Q` для "Турбо-сайт").

Наблюдатели выставляются отдельным шагом через D7-объект лида:

- `\Bitrix\Crm\Item\Lead::createInstance($leadId)`
- `$item->setObservers($observerIds)`
- `$item->save()`

Чтобы не терялись роботы и БП при создании из PHP-скрипта, добавлен явный автозапуск:

- `\Bitrix\Crm\Automation\Starter(\CCrmOwnerType::Lead, $leadId)->runOnAdd()`
- `\CCrmBizProcHelper::AutoStartWorkflows(\CCrmOwnerType::LeadName, $leadId, \CCrmBizProcEventType::Create, $errors)`

### 2) Работа с данными списков/инфоблоков

Используется внутренний API инфоблоков:

- `CIBlockElement::GetList(...)` — поиск элемента в ИБ 54 по названию,
- `CIBlockElement::GetProperty(...)` — чтение свойств (источник, город, исполнитель и т.д.).

### 3) Уведомления в чат

Используется IM модуль коробки:

- `CIMMessenger::Add(...)`.

## Методы и точки интеграции (по файлам)

### `deal_webhook_handler.php`

Основной обработчик интеграции:

- входящий webhook и разбор payload,
- REST-вызовы в облако:
  - `crm.deal.get`
  - `crm.contact.get`
  - `crm.deal.fields`
- построение полей лида,
- создание лида через внутренний CRM API коробки,
- явный запуск роботов и бизнес-процессов после `Add`,
- ведение состояния в JSON-файлах,
- логирование и уведомления,
- action `create_ib54_from_deal`: по ссылке из чата проверяет, есть ли элемент в списке 54; если уже есть — показывает сообщение "Такой элемент уже добавлен" и ссылку на открытие, если нет — создает элемент (с источником "Турбо-сайт") и открывает карточку по URL вида `/workgroups/group/1/lists/54/element/0/{id}/?list_section_id=`.

### `bitrix_init.php`

Минимальная webhook-only инициализация ядра:

- определение `DOCUMENT_ROOT`,
- подключение `prolog_before.php`,
- подключение обязательных модулей `crm` и `iblock`.

### `deal_webhook_config.php`

Конфигурация проекта, вынесенная из кода:

- webhook URL/токен облака,
- ID инфоблоков и поля лида,
- настройки уведомлений (чат, шаблоны сообщений, ссылки),
- шаблон ссылки на лид коробки `notifications.lead_detail_url_template`,
- параметры логирования и путь к файлу маппинга.

В проекте используется только webhook-поток без агентного и таймерного контуров.

## Хранилища состояния и служебные файлы

- `deal_lead_pairs.json` — карта соответствий `deal_id -> lead_id`.
- `var/deal_to_lead_webhook.log` — журнал обработки и ошибок.

### Мониторинг и ручной reprocess

- Мониторинг ошибок выполняется по `var/deal_to_lead_webhook.log`.
- При сбое отдельной сделки используется повторная отправка webhook с тем же `deal_id`.
- Идемпотентность обеспечивается `deal_lead_pairs.json`.
- При записи `deal_lead_pairs.json` используется lock (`.lock` файл + `flock`), что снижает риск потери данных при параллельных webhook.

## Технический стек API внутри проекта

- **Legacy API Bitrix (основной рабочий слой для CRM/iblock в проекте):**
  - `CCrmLead`, `CCrmBizProcHelper`, `CIBlockElement`, `CModule`, `CIMMessenger`.
- **D7 API (точечно):**
  - `\Bitrix\Main\Loader`,
  - `\Bitrix\Crm\Item\Lead` (наблюдатели),
  - `\Bitrix\Crm\Automation\Starter` (запуск роботов).
- **Bitrix24 Cloud REST:**
  - POST через `curl` к webhook endpoint (`/rest/.../<method>`).

---

## Важный вывод по архитектуре

Проект реализован как **мост "облако -> коробка"**:

- чтение и метаданные берутся через облачный REST,
- запись сущностей и бизнес-обогащение выполняются внутренними методами коробки.

То есть используются сразу три слоя: **REST + внутренний legacy API + частичный D7 API**.
