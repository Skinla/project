# Управление очередями обработки

## Описание системы

Система управления очередями обработки предназначена для мониторинга и управления файлами на разных этапах обработки лидов. Веб-интерфейс доступен по адресу: `retry-raw-errors.php`

## Структура очередей

Система использует конвейерную обработку данных через несколько очередей:

```
Raw → Detected → Normalized → Processed
                ↓
            Duplicates (если дубль)
                ↓
            Failed (если ошибка)
                ↓
            raw_errors (если критическая ошибка)
```

## Описание вкладок

### 1. Ошибки (raw/raw_errors)

**Что лежит:** Файлы, которые не удалось обработать на этапе определения типа данных.

**Формат файлов:** `raw_*.json`

**Причины попадания в ошибки:**
- Некорректный JSON
- Неизвестный домен (не найден в ИБ54)
- Нормализатор не найден
- Пустые/тестовые запросы

**Содержимое:** Сырые данные с ошибками (raw_body, raw_headers, parsed_data, source_domain)

**Действия:**
- **Обработать** — переместить файл в `raw` для повторной обработки
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

**Глобальные действия:**
- **Обработать все файлы** — переместить все файлы из `raw_errors` в `raw` и запустить обработку
- **Запустить обработку очередей** — запустить полный цикл обработки всех очередей

---

### 2. Raw

**Что лежит:** Сырые данные, только что поступившие из webhook'ов.

**Формат файлов:** `raw_*.json`

**Содержимое:** Исходные данные запроса:
- `raw_body` — тело запроса (URL-encoded или JSON)
- `raw_headers` — HTTP заголовки (HTTP_REFERER, HTTP_ORIGIN, QUERY_STRING и т.д.)
- `parsed_data` — распарсенные данные формы
- `source_domain` — домен источника
- `timestamp` — время создания

**Статус:** Ожидают определения типа данных и выбора нормализатора

**Действия:**
- **Обработать** — запустить обработку очередей (raw → detected → normalized → processed)
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

### 3. Detected

**Что лежит:** Файлы с определенным типом данных и выбранным нормализатором.

**Формат файлов:** `detected_*.json`

**Содержимое:** Данные из `raw` + дополнительная информация:
- `normalizer_file` — выбранный нормализатор (например, `tilda_normalizer.php`)
- `detected_at` — время определения типа
- `source_domain` — домен источника
- `raw_data` — полные исходные данные
- `parsed_data` — распарсенные данные
- `raw_body`, `raw_headers` — исходные данные запроса

**Статус:** Ожидают нормализации данных

**Действия:**
- **Обработать** — запустить нормализацию (detected → normalized)
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

### 4. Normalized

**Что лежит:** Файлы с нормализованными данными в стандартном формате.

**Формат файлов:** `normalized_*.json`

**Содержимое:** Стандартизированные данные:
- `phone` — телефон (только цифры)
- `name` — имя клиента
- `email` — email (если есть)
- `source_domain` — домен источника
- `utm_source`, `utm_medium`, `utm_campaign`, `utm_term` — UTM метки
- `form_name` — название формы
- `comment` — комментарий с дополнительными данными
- `timestamp` — время нормализации
- `raw_data` — исходные данные

**Статус:** Готовы к созданию лидов в Bitrix24

**Действия:**
- **Обработать** — запустить создание лидов (normalized → processed)
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

### 5. Processed

**Что лежит:** Успешно обработанные файлы (лиды созданы в Bitrix24).

**Формат файлов:** `processed_*.json`

**Содержимое:** Нормализованные данные + информация о созданном лиде:
- Все поля из `normalized` +
- `lead_id` — ID созданного лида в Bitrix24
- `processed_at` — время обработки

**Статус:** Финальный, обработка завершена успешно

**Действия:**
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

### 6. Duplicates

**Что лежит:** Файлы, которые были определены как дубликаты.

**Формат файлов:** Файлы из `normalized`, перемещенные в `duplicates`

**Содержимое:** Нормализованные данные, которые уже были обработаны ранее

**Причины дубликатов:**
- `already_processed` — телефон + домен уже обработаны ранее
- Дубликат обнаружен на этапе создания лида

**Действия:**
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

### 7. Failed

**Что лежит:** Файлы, которые не удалось обработать на этапе создания лидов.

**Формат файлов:** `failed_*.json`

**Содержимое:** Нормализованные данные, для которых не удалось создать лид

**Причины ошибок:**
- Ошибка API Bitrix24
- Некорректные данные
- Ошибка подключения к БД
- Ошибка поиска элемента в ИБ54

**Действия:**
- **Обработать** — переместить файл в `raw` для повторной обработки с начала
- **Просмотр** — посмотреть содержимое файла
- **Удалить** — удалить файл

---

## Процесс обработки

### Этап 1: Raw → Detected

**Функция:** `processRawFiles()` в `data_type_detector.php`

**Что происходит:**
1. Читаются файлы из папки `raw/`
2. Определяется тип данных (по домену или formid)
3. Выбирается нормализатор из ИБ54
4. Файл перемещается в `detected/` с информацией о нормализаторе

**Логика поиска нормализатора:**
1. Сначала ищется по `source_domain` в ИБ54 (по полю `NAME`)
2. Если не найден и есть `formid` → ищется по `formid` в ИБ54
3. Если оба не найдены → файл перемещается в `raw_errors`

**Особенности:**
- Обрабатывается максимум 50 файлов за запуск
- Таймаут: 25 секунд на весь цикл
- Пустые/тестовые запросы перемещаются в `raw_errors` без уведомлений

---

### Этап 2: Detected → Normalized

**Функция:** `processDetectedFiles()` в `queue_manager.php`

**Что происходит:**
1. Читаются файлы из папки `detected/`
2. Загружается нормализатор по `normalizer_file`
3. Вызывается метод `normalize()` нормализатора
4. Данные преобразуются в стандартный формат
5. Файл перемещается в `normalized/`

**Особенности:**
- Обрабатывается максимум 50 файлов за запуск
- Таймаут: 25 секунд на весь цикл
- Используются `.lock` файлы для предотвращения параллельной обработки
- При ошибке файл копируется в `queue_errors`

---

### Этап 3: Normalized → Processed

**Функция:** `processNormalizedFiles()` в `lead_processor.php`

**Что происходит:**
1. Читаются файлы из папки `normalized/`
2. Проверяется дубликат (телефон + домен)
3. Если не дубль:
   - Ищется элемент в ИБ54 по `source_domain`
   - Получаются настройки (исполнитель, инфоповод, наблюдатели)
   - Создается лид в Bitrix24 через REST API
   - Файл перемещается в `processed/`
4. Если дубль → файл перемещается в `duplicates/`

**Особенности:**
- Обрабатывается максимум 50 файлов за запуск
- Таймаут: 25 секунд на весь цикл
- Используются `.lock` файлы для предотвращения параллельной обработки
- При ошибке файл перемещается в `failed/`

---

## Использование интерфейса

### Просмотр файла

1. Выберите нужную вкладку (Ошибки, Raw, Detected и т.д.)
2. Найдите нужный файл в списке
3. Нажмите кнопку **"Просмотр"**
4. Содержимое файла отобразится прямо под названием файла
5. Нажмите **"Просмотр"** еще раз, чтобы скрыть содержимое

### Обработка файла

1. Выберите файл в нужной вкладке
2. Нажмите кнопку **"Обработать"**
3. Файл будет обработан согласно логике для данной очереди:
   - **Raw/Detected/Normalized** → запускается обработка очередей
   - **Failed** → файл перемещается в `raw` для повторной обработки
   - **Ошибки** → файл перемещается в `raw` для повторной обработки

### Удаление файла

1. Выберите файл в нужной вкладке
2. Нажмите кнопку **"Удалить"**
3. Подтвердите удаление
4. Файл и связанный `.lock` файл (если есть) будут удалены

### Глобальные действия

**Обработать все файлы** (только вкладка "Ошибки"):
- Перемещает все файлы из `raw_errors` в `raw`
- Запускает обработку очередей

**Запустить обработку очередей**:
- Запускает полный цикл обработки всех очередей
- Обрабатывает файлы из `raw`, `detected`, `normalized`

---

## Статистика

В верхней части страницы отображается статистика по всем очередям:
- **RAW** — количество файлов в очереди `raw`
- **DETECTED** — количество файлов в очереди `detected`
- **NORMALIZED** — количество файлов в очереди `normalized`
- **PROCESSED** — количество успешно обработанных файлов
- **DUPLICATES** — количество дубликатов
- **FAILED** — количество файлов с ошибками

---

## Ограничения и защита от зависаний

Система имеет встроенные механизмы защиты от зависаний:

1. **Rate Limiting:**
   - Максимум 50 файлов обрабатывается за один запуск
   - Предотвращает перегрузку системы

2. **Таймауты:**
   - Максимум 25 секунд на весь цикл обработки
   - Предотвращает долгие зависания

3. **Файловые блокировки:**
   - Используются `.lock` файлы для предотвращения параллельной обработки
   - Автоматическое удаление устаревших блокировок (старше 5 минут)

4. **Неблокирующие операции:**
   - Используется `flock()` с флагом `LOCK_NB` (non-blocking)
   - При невозможности получить блокировку операция пропускается

## Настройка нормализаторов в ИБ54

Для корректной работы системы необходимо настроить элементы в инфоблоке 54:

### По домену:
- **NAME** = домен (например, `"medtour-ul.tilda.ws"`)
- **PROPERTY_388** (Обработчик) = имя нормализатора (например, `"tilda_normalizer.php"`)

### По formid (для форм Tilda):
- **NAME** = formid (например, `"form838072035"`)
- **PROPERTY_388** (Обработчик) = имя нормализатора (например, `"tilda_normalizer.php"`)

### Доступные нормализаторы:
- `tilda_normalizer.php` — для форм Tilda
- `calltouch_normalizer.php` — для CallTouch
- `bitrix24_webhook_normalizer.php` — для Bitrix24 webhook'ов
- `generic_normalizer.php` — универсальный нормализатор
- `koltaсh_normalizer.php` — для Koltaсh

## Логирование

Все операции логируются в файлы:
- Глобальный лог: `logs/global.log`
- Логи обработки очередей: `logs/queue_processing.log`
- Логи ошибок: `logs/error.log`

---

## Уведомления в чат

Система автоматически отправляет уведомления в Bitrix24 чат при:
- Накоплении 10+ файлов в очереди `raw`
- Критических ошибках обработки
- Ошибках поиска нормализатора в ИБ54

Уведомления не отправляются для:
- Пустых/тестовых запросов
- Обычных дубликатов

## Технические детали

### Структура файлов

Все файлы хранятся в формате JSON в соответствующих папках:
- `queue/raw/` — сырые данные
- `queue/detected/` — определенные типы
- `queue/normalized/` — нормализованные данные
- `queue/processed/` — обработанные файлы
- `queue/duplicates/` — дубликаты
- `queue/failed/` — ошибки
- `queue/raw/raw_errors/` — критические ошибки
- `queue/queue_errors/` — ошибки нормализации

### Формат имен файлов

- `raw_YYYYMMDD_HHMMSS_XXXXXXXX.json`
- `detected_YYYYMMDD_HHMMSS_XXXXXXXX.json`
- `normalized_YYYYMMDD_HHMMSS_XXXXXXXX.json`
- `processed_YYYYMMDD_HHMMSS_XXXXXXXX.json`

Где `XXXXXXXX` — случайный хеш для уникальности.

## Решение проблем

### Файлы застряли в очереди

1. Проверьте наличие `.lock` файлов
2. Удалите устаревшие `.lock` файлы (старше 5 минут)
3. Нажмите **"Запустить обработку очередей"**

### Файлы не обрабатываются

1. Проверьте логи в `logs/global.log`
2. Убедитесь, что элемент есть в ИБ54 с правильным нормализатором
3. Проверьте, что нормализатор существует в папке `normalizers/`

### Много файлов в raw_errors

1. Проверьте, какие домены/formid не найдены в ИБ54
2. Добавьте недостающие элементы в ИБ54
3. Используйте **"Обработать все файлы"** для повторной обработки

---

## Контакты и поддержка

При возникновении проблем проверьте:
1. Логи в `logs/global.log`
2. Настройки в ИБ54
3. Доступность Bitrix24 REST API
4. Корректность путей к файлам и папкам
