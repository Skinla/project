# Отчёт о расследовании инцидента: 1100 дубликатов лидов с телефоном +7 905 012-57-53

**Дата инцидента:** 13 марта 2026  
**Телефон:** +7 905 012-57-53  
**Количество дубликатов:** ~1100 лидов  
**Источник данных:** Bitrix24 Webhook (портал b24-gyahw8.bitrix24.ru)

---

## 1. Идентификация источника

Все проблемные запросы пришли из **Bitrix24 Webhook**:
- **Портал:** b24-gyahw8.bitrix24.ru (auth.domain)
- **Документ:** document_id = LEAD_365978 (один и тот же лид)
- **Форма:** AllDent - allon4 - Нижний Новгород
- **Клиент:** Александр, +7 (905) 012-57-53

Данные в `queue/queue_errors/` показывают идентичную структуру запросов с `document_id[2]=LEAD_365978`.

---

## 2. Корневая причина (по результатам анализа логов)

### 2.1. Баг в DuplicateChecker — записи с lead_id=null не загружаются

**Критическая находка в `duplicate_checker.php`:**

При ошибке создания лида вызывается `markAsProcessed($duplicateKey, null)`. В файл записывается строка:
```
[2026-03-13 12:40:02] phone:+7 (905) 012-57-53|AllDent - allon4 - Нижний Новгород | lead_id:
```

Но `loadProcessedPhones()` использует regex с **обязательным** `lead_id:(\d+)`:
```php
preg_match('/.*lead_id:(\d+)/', $line, $matches)
```

Запись с пустым или `null` lead_id **не совпадает** с regex и **не загружается**. В результате при следующем запуске DuplicateChecker не видит этот телефон+домен как уже обработанный → запрос снова идёт на создание лида → снова ошибка → цикл повторяется.

### 2.2. createLead постоянно возвращал null для этого запроса

**Данные из логов:**
- Raw-файл `raw_20260312_020518_687e47f9d76ee.json` создан **12 марта 02:05:18**
- В `queue_errors` — **45+ normalized-файлов** с тем же `raw_file_path`, с 11:40 до 12:51 **13 марта**
- Все они ссылаются на один и тот же raw-файл

Файлы попадают в `queue_errors` только при `copyToQueueErrors()`, то есть когда **createLead возвращает null**. Значит, для этого запроса создание лида каждый раз завершалось ошибкой (API, таймаут, валидация и т.п.). Точная причина не видна в текущих логах (global.log обрезан с 16:25).

### 2.3. Raw-файл не удаляется при ошибке

При ошибке создания лида (`leadId === null`) raw-файл **не удаляется** — удаление выполняется только в блоке успешного создания. В сочетании с п. 2.1 и 2.2 raw-файл остаётся в очереди и обрабатывается снова при каждом запуске.

### 2.4. Цепочка причин

1. **createLead** для этого запроса возвращает null (ошибка API/таймаут и т.д.)
2. Вызывается **markAsProcessed(key, null)** → в лог пишется `lead_id:` (пусто)
3. **loadProcessedPhones** не загружает эту запись из-за regex `lead_id:(\d+)`
4. Raw-файл **не удаляется** при ошибке
5. При следующем запуске raw снова обрабатывается → снова ошибка → цикл
6. Ручная очистка raw прерывает цикл

## 4. Рекомендации по исправлению

### 4.1. Критическое исправление — DuplicateChecker

**Исправить `loadProcessedPhones()` в `duplicate_checker.php`:**

Текущий regex не загружает записи с `lead_id:null` или пустым lead_id. Нужно либо:
- изменить regex, чтобы допускать пустой lead_id: `lead_id:(\d*|null)?`,  
- либо при `markAsProcessed` с null не сохранять в processed (оставить только в processing до истечения таймаута),  
- либо использовать отдельный формат/файл для «неудачных» обращений и учитывать их при проверке дубликатов.

Цель: при следующем запуске запрос с тем же телефон+домен, по которому уже была ошибка, должен распознаваться как дубликат и не отправляться на повторное создание лида.

### 4.2. Срочные меры

1. **Удалять raw при ошибке создания лида**  
   При `copyToQueueErrors` (когда createLead вернул null) тоже удалять raw-файл по `raw_file_path`, чтобы он не застревал в очереди.

2. **Удалять raw при moveToDuplicates**  
   При обнаружении дубликата удалять raw по `raw_file_path` из normalizedData.

3. **Добавить блокировку в cron**  
   В `queue_manager.php` при `run` использовать тот же `webhook.lock`, что и вебхук.

### 4.3. Долгосрочные меры

1. **Увеличить окно DuplicateChecker** до 24 часов.
2. **Добавить проверку по document_id** для Bitrix24.
3. **Мониторинг застрявших raw-файлов** — уведомление, если файлы в raw/ старше N часов.

## 6. Итог

**Основная причина застревания:** Комбинация трёх факторов:

1. **createLead возвращал null** для запроса `raw_20260312_020518_687e47f9d76ee.json` (ошибка API/таймаут и т.п.), из‑за чего файл попадал в `queue_errors`.
2. **Баг в DuplicateChecker:** `loadProcessedPhones()` не загружает записи с `lead_id:null` из‑за regex `lead_id:(\d+)`, поэтому при следующем запуске тот же запрос не считался дубликатом и снова обрабатывался.
3. **Raw-файл не удаляется при ошибке** — удаление выполняется только при успешном создании лида.

В результате raw-файл оставался в очереди, при каждом запуске создавался лид с ошибкой, запись дубликата не сохранялась, и цикл повторялся до ручной очистки raw.

**Подтверждение:** Эффект прекратился после принудительной очистки всех файлов в `queue/raw/`. В `queue_errors` есть 45+ файлов с одним и тем же `raw_file_path`.

**Приоритет:** Исправить DuplicateChecker (загрузка записей с lead_id=null) и удалять raw-файл при ошибке и при дубликате.
